## ADDED Requirements

### Requirement: Streaming CSV import for Ingredients

The system SHALL accept a multipart-uploaded CSV file at `POST /ingredients/import` and produce a structured outcome for the caller.

#### Scenario: dry-run preview against a clean CSV

- WHEN a Manager uploads a 10-row CSV with all required columns and valid values
- AND the request includes `?dryRun=true`
- THEN the response is `200 OK` with body `{ valid: 10, invalid: 0, errors: [] }`
- AND no `Ingredient` rows are written to the database

#### Scenario: dry-run preview against a CSV with mixed validity

- WHEN a Manager uploads a 100-row CSV where 95 rows are valid and 5 rows have errors (missing `categoryName`, unknown `baseUnitType`, negative `densityFactor`, etc.)
- AND the request includes `?dryRun=true`
- THEN the response body has `{ valid: 95, invalid: 5, errors: [<5 entries with rowIndex, column, code, message>] }`
- AND no rows are written

#### Scenario: commit path persists valid rows and surfaces errors

- WHEN a Manager uploads the same mixed CSV with `?dryRun=false`
- THEN 95 `Ingredient` rows are persisted under the caller's organizationId
- AND the response body is `{ valid: 95, invalid: 5, errors: [...] }` with the same 5 errors
- AND each persisted row has its `internalCode` autogenerated when blank in the CSV

#### Scenario: chunked transaction semantics on mid-import failure

- WHEN a 1000-row CSV has a row at index 750 that triggers a database CHECK constraint failure
- AND the chunk size is 500
- THEN chunks 1 (rows 1-500) commit
- AND chunk 2 (rows 501-1000) rolls back atomically
- AND the response body reports 500 valid + 500 marked errored from chunk 2

#### Scenario: 10k rows commit in <60 seconds (NFR)

- WHEN a Manager uploads a 10k-row CSV with all valid rows on the CI runner
- AND `?dryRun=false`
- THEN all 10k rows persist in <60 seconds wall-clock

#### Scenario: STAFF cannot import

- WHEN a STAFF user submits any import request
- THEN the response is `403 Forbidden` with `{ code: 'INSUFFICIENT_ROLE' }`

### Requirement: CSV column schema

The system SHALL parse CSV files conforming to a documented column schema.

#### Scenario: required columns present

- WHEN a CSV header row includes `name`, `categoryName`, `baseUnitType`
- THEN parsing proceeds and rows are validated against the schema

#### Scenario: required column missing

- WHEN any of `name`, `categoryName`, `baseUnitType` is absent from the header
- THEN the response is `400 Bad Request` with `{ code: 'CSV_IMPORT_INVALID_FORMAT', detail: '<missing column>' }`
- AND no rows are processed

#### Scenario: optional columns absent

- WHEN `internalCode`, `densityFactor`, or `notes` columns are absent
- THEN parsing proceeds; missing optional fields are treated as not-provided
- AND a blank `internalCode` cell triggers autogeneration via `Ingredient.create()`

#### Scenario: categoryName resolution unique match

- WHEN exactly one category in the org matches a row's `categoryName` (case-insensitive)
- THEN the row's `categoryId` is set to that category's id

#### Scenario: categoryName resolution miss

- WHEN no categories match a row's `categoryName`
- THEN the row is invalid with `{ code: 'CATEGORY_NOT_FOUND' }`

#### Scenario: categoryName resolution ambiguous

- WHEN multiple categories share the same `name` under different parents
- THEN the row is invalid with `{ code: 'CATEGORY_AMBIGUOUS_NAME', hint: 'Use slug path (e.g. dry-pantry/oils-vinegars)' }`

#### Scenario: slug-path category resolution

- WHEN a `categoryName` cell contains `/` (e.g. `dry-pantry/oils-vinegars`)
- THEN the resolver walks the path: root match → child match
- WHEN any segment fails
- THEN the row is invalid with `{ code: 'CATEGORY_NOT_FOUND' }`

### Requirement: Streaming CSV export for Ingredients

The system SHALL stream the org's ingredient list as CSV at `GET /ingredients/export.csv`.

#### Scenario: full export of a small org

- WHEN a Manager requests `GET /ingredients/export.csv?organizationId=<uuid>` for an org with 50 active ingredients
- THEN the response is `200 OK` with `Content-Type: text/csv; charset=utf-8`
- AND `Content-Disposition: attachment; filename="ingredients-<org>-<YYYY-MM-DD>.csv"`
- AND the body is a CSV with the canonical header row + 50 data rows

#### Scenario: export of an org with 10k rows streams without loading into memory

- WHEN a Manager requests the export for an org with 10k active ingredients
- THEN the response begins streaming within 1s
- AND the export service paginates via `IngredientRepository.pageByOrganization` (cursor-based) so heap usage stays bounded regardless of total row count

#### Scenario: STAFF cannot export

- WHEN a STAFF user submits an export request
- THEN the response is `403 Forbidden` with `{ code: 'INSUFFICIENT_ROLE' }`

#### Scenario: export header round-trips through import

- WHEN the export of an org's ingredients is fed back into the import endpoint with `dryRun=true`
- THEN every row validates as `valid` (no schema errors)
- AND the count of `valid` equals the original ingredient count (modulo any soft-deleted rows excluded by export)
