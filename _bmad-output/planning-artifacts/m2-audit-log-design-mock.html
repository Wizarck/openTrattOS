m2-audit-log Wave 1.9 design mock · 2026-05-06 · pre-implementation review

Single canonical audit_log table extracts the cache+audit pattern that 5 bounded contexts currently re-implement in their own per-BC tables and jsonb columns. @OnEvent subscriber persists a row per emitted event. Per-BC audit columns survive this slice (cleanup deferred to m2-audit-log-cleanup).

One subscriber, eight event types, no per-BC import of AuditLogService. Subscriber crashes are caught + logged; emitter never blocks.

Design invariant: BC services never import AuditLogService. They emit events; the subscriber persists. Adding a new event type in a future BC = 1 line of @OnEvent in the subscriber + a constants entry. No table or controller change.

Indexes (3 — write-heavy table, minimise overhead)

  • (organization_id, aggregate_type, aggregate_id, created_at DESC) drill-down by entity history
  • (organization_id, event_type, created_at DESC) global filter by event type
  • (organization_id, actor_user_id, created_at DESC) WHERE actor_user_id IS NOT NULL partial — user history

4 · AuditEventEnvelope shape — sample emit

Existing emitters publish ad-hoc payload objects today. This slice migrates them to a typed envelope so the subscriber can map without per-event-type translation logic.

// apps/api/src/audit-log/application/types.ts
export interface AuditEventEnvelope<TBefore = unknown, TAfter = unknown> {
  organizationId: string;
  aggregateType: string;          // 'recipe' | 'ingredient' | 'ai_suggestion' | ...
  aggregateId: string;
  actorUserId: string | null;     // null for system events
  actorKind: 'user' | 'agent' | 'system';
  agentName?: string;             // populated when actorKind === 'agent'
  payloadBefore: TBefore | null;
  payloadAfter: TAfter | null;
  reason?: string;                // overrides + rejections
  citationUrl?: string;           // AI events
  snippet?: string;               // AI events
}

// emit example — ai-suggestions.service.accept()
this.events.emit(AuditEventType.AI_SUGGESTION_ACCEPTED, {
  organizationId: orgId,
  aggregateType: 'ai_suggestion',
  aggregateId: suggestion.id,
  actorUserId: userId,
  actorKind: 'user',
  payloadBefore: { status: 'pending', value: suggestion.suggestedValue },
  payloadAfter:  { status: 'accepted', acceptedValue: suggestion.acceptedValue ?? suggestion.suggestedValue },
  citationUrl: suggestion.citationUrl,
  snippet: suggestion.snippet,
} satisfies AuditEventEnvelope);

8 · Out of scope (filed as follow-ups)

  • m2-audit-log-cleanup — drop redundant per-BC audit columns + tables (recipe_cost_history, ingredients.overrides jsonb, recipes.allergens_overrides jsonb).
  • m2-audit-log-export — GET /audit-log/export.csv for offline analysis.
  • m2-audit-log-fts — Postgres FTS index on payload_before/after + reason + snippet for cross-BC search by free text.
  • m2-audit-log-retention — archival policy (partitioning by month, cold storage after N years).

9 · Risks acknowledged

  • Subscriber crash swallows audit events — try/catch + log; emitter never blocks. Dead-letter queue is a future slice if reliability becomes a concern.
  • High write volume — single row + 3 index updates ≤ 1ms. Storage growth ~22 GB/mes/heavy-org → partitioning when justified (years out).
  • Polymorphic FK — aggregate_id has no real FK; aggregate validity guaranteed by emitter (entity exists when event fires).
  • Trade-off: per-BC audit columns survive — two sources of truth for one release. Canonical path = audit_log; legacy columns deprecated-but-readable until cleanup slice.
