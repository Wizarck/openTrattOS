# J11 — Purchasing Manager runs the procurement loop

The procurement loop spans three sub-surfaces on one tabbed screen: **POs** (place orders ahead of delivery), **Goods Receipts** (confirm what arrived against what was ordered), and **Reconciliation** (resolve discrepancies before payment). Andoni works POs at his desk, GRs at the receiving dock on the tablet, and reconciliations back at the desk. The visual surface is [mock-j11-procurement.html](variants/mock-j11-procurement.html); this doc anchors the three views.

## Goal

In a 5-minute window before Friday delivery, Andoni (i) creates POs for the weekend menu, (ii) at the dock taps GR rows as they arrive (or — more common — they auto-arrive from Hermes invoice-photo ingestion, [J12](j12.md)), and (iii) reconciles any quantity / price discrepancy against the supplier. The loop produces lots that the rest of M3 reads.

## Trigger

Andoni opens `/procurement` from the top nav, which lands him on the POs tab by default. Tablet at the dock deep-links to `/procurement/goods-receipts?location=bilbao&pending=true` — the pre-filter narrows the surface to "today's pending deliveries here".

## Walkthrough

The numbers below correspond to the regions of [mock-j11-procurement.html](variants/mock-j11-procurement.html).

1. **Tab strip (sticky).** Three tabs: `Órdenes de compra` (12) · `Recepciones` (3 pendientes) · `Reconciliación` (2 abiertas). Counters update live. The tab strip is the only nav element below the global breadcrumb.
2. **POs view — table-first.** Plain table, no card. Columns: PO# · proveedor · fecha · líneas · total · estado (`borrador` / `enviada` / `parcialmente recibida` / `cerrada` / `cancelada`). Rows sorted by `enviada DESC` (recent first). Action column has a single ghost: `ver →`. Above the table: a primary `--accent` CTA `Nueva OC`. Filters chip group above the CTA: location (multi-select) · proveedor (multi-select) · estado (single).
3. **PO detail (inline drawer or full page).** Click on a row → drawer mounts on right, full-page on phone. Shows: header (proveedor + dirección + fecha + nº interno) · líneas table (producto · cantidad · unidad · precio unit · subtotal) · footer (subtotal · IVA · total · términos). For drafts: edit-in-place. For sent/closed: read-only. A `Cancelar OC` ghost on drafts; `Cerrar OC` ghost on partially-received.
4. **Goods Receipts view — line-by-line.** Default filter = `pendientes`. List view: each row = one PO line that arrived. Columns: PO# · producto · cantidad esperada · cantidad recibida (editable) · lote (auto-generated) · fecha caducidad · acción. Andoni at the dock taps the row, confirms the quantity matches (or enters the actual quantity if not), confirms or edits the supplier-provided lot code, enters expiry from the package, taps `Confirmar`. Each confirmation writes a `goods_receipt` row + a new `Lot` per ADR-031 indexing. Bulk-confirm CTA at top: `Confirmar todo lo que coincida (8)` — saves dock time when every line matches.
5. **GR detail (per line).** Tap → drawer mounts. Shows: PO line context · supplier-provided fields · auto-generated lot code (editable with confirmation) · expiry date picker · `attachments` (links to photos from J12 if invoice-photo ingest pre-created the GR). A mute eyebrow at the top reads `Pre-cargado por Hermes desde foto · 14:08 CEST · revisar →` if Hermes seeded it.
6. **Reconciliation view.** Default filter = `abiertas`. List view: each row = a GR that didn't fully match its PO. Columns: PO# · proveedor · tipo de discrepancia (`cantidad` / `precio` / `producto` / `lote-no-conforme`) · diff · estado de resolución · acción. Tap → drawer with the side-by-side comparison (PO expected vs GR actual). Resolution actions: `Aceptar diferencia` (writes audit_log; closes), `Solicitar nota de crédito` (links to supplier email template), `Devolver` (creates a Goods Return draft — out of scope for J11 mock but linked).
7. **Audit chip on every drawer.** Every row's drawer carries a `--mute` link at the bottom: `audit_log AL-2026-NNNNNN · ver chain →`. Procurement is a financial domain — every action is queryable. The link routes to [J12](j12.md) (audit-log browse extension) filtered to this aggregate.

## Edge cases

- **Supplier sent product NOT on the PO.** GR line carries a `producto no en OC` status; the line cannot be confirmed without operator decision. Two paths: (i) `Aceptar y añadir como línea adicional` (mutates the PO with audit_log envelope), or (ii) `Devolver al proveedor`. Both write rows; no silent acceptance.
- **Partial delivery (PO line received in two shipments).** First GR confirms a partial quantity; PO state goes to `parcialmente recibida`. Second GR closes the line; PO transitions to `cerrada` only when every line is closed.
- **Lot code from supplier conflicts with a previously-received lot.** System prepends an internal `INT-` prefix and surfaces a mute eyebrow `Lote duplicado · usaremos INT-{supplier_code} para evitar colisión`. The original supplier code is preserved in the GR row's metadata.
- **Reconciliation requires Owner sign-off.** Per RBAC (ADR-006 inheritance): discrepancies > €500 OR involving `lote-no-conforme` require an Owner approval gate. The action button reads `Solicitar aprobación Owner` instead of resolving directly. The Owner receives an email with deep-link back to the reconciliation drawer.
- **Hermes invoice-photo pre-seeded the GR but vision-LLM confidence was low (<0.60).** The GR appears in the dock list with a `--destructive` eyebrow `Confianza baja · revisar manualmente · 3 campos sin extraer`. Andoni completes the missing fields by hand. Per ADR-034 confidence-banded HITL — `< 0.60` always routes to human review.
- **Offline mode at the dock.** The tablet GR flow caches locally; confirmations queue and replay on reconnect. A persistent banner at the top of the GR view reads `Modo offline · 4 confirmaciones en cola` when applicable.
- **Reconciliation aborted mid-resolution.** Drafts persist locally for 24 hours; the row in the list shows a mute eyebrow `Borrador de resolución · 14:32 ayer`. Operator resumes by tapping.
- **PO referencing a deleted product.** Soft-delete preserves the reference (ADR-009); the line carries a `producto archivado` mute eyebrow. New POs cannot select the product.

## Decisions specific to J11

- **One screen, three tabs — not three routes.** The procurement loop is a workflow, not three unrelated views. A single URL with tab state keeps the operator's mental model contiguous; opening a discrepancy from the dock view lands directly in reconciliation without a context switch. The route /procurement preserves the active tab in the URL for deep-linking.
- **Bulk-confirm "todo lo que coincida" CTA.** Dock time is precious. If 8 of 10 lines match the PO exactly (quantity + lot pattern + expiry sane), one tap confirms them; the remaining 2 stay open for individual handling. The `coincida` predicate is documented in implementation notes — never accept silent matches the operator can't audit later.
- **Lot code is editable on GR confirmation but with confirmation.** The supplier-provided code is the default; if the operator overrides, a small confirm `¿Sobrescribir lote del proveedor?` appears. This protects against a typo that breaks the recall chain — the recall investigation surface (J6) queries by lot code, so a wrong code at this step is a hidden cost later.
- **Reconciliation resolution writes an audit_log row, never overwrites the GR.** The original GR remains immutable. The resolution is a forward-extending event. (Same principle as J7 dossier addendum — chain of custody.)
- **Owner-approval gate for high-value or non-conformity discrepancies.** The Manager has procurement authority, but the gate protects against fraud and accidents. The threshold is configurable per organization. (RBAC ADR-006 + capability flags inheritance.)
- **Hermes-seeded GRs are visually distinguished but not treated specially.** The mute eyebrow `Pre-cargado por Hermes …` is informational; the GR is still subject to operator confirmation. Auto-confirming on high-confidence would conflict with the EU AI Act HITL posture (ADR-034 + PRD-M3 §EU AI Act). Operator review is the system's stance, even when the vision-LLM is confident.
- **Audit chip on every drawer.** Procurement is the most fraud-adjacent domain in M3. Surfacing the audit_log link on every drawer trains the operator that everything is queryable — and surfaces the forensic affordance for inspectors during J9 export reads.
- **Filters above the CTA, not in a separate panel.** The chip group is sticky-context: the operator typically filters before creating a new PO ("for Bilbao only"). Putting the filters above the primary CTA reinforces that mental model and removes a click.
- **Tablet GR view uses ≥ 48 px touch targets across the board; row height ≥ 64 px.** Dock conditions = wet hands, gloves, glance scans. Same kitchen-real principle as J10 inherited.

## Notes for implementation

- The three tabs share a single route component; tab state in URL query (`?tab=po|gr|recon`). React Suspense streams the active tab's payload.
- GR confirmation calls `procurement.create-goods-receipt` MCP capability per ADR-038 namespacing. Lot creation cascades atomically in the same transaction (per ADR-031).
- The "matches PO exactly" predicate for bulk-confirm: `quantity_diff == 0 AND lot_code matches supplier_pattern AND expiry_within_supplier_typical_range AND no_status_flag`. Documented as `BULK_CONFIRM_PREDICATE` constant.
- Reconciliation resolution writes `procurement.resolve-reconciliation` envelope with `resolution_type` + `notes` + `linked_credit_note_url` (nullable). No PO mutation; the audit_log row is the resolution.
- Owner-approval threshold reads from `organizations.procurement_approval_threshold_eur` column (default €500). NULL = no gate.
- Hermes-seeded GR rows carry `metadata.source = 'hermes-invoice-photo'` and `metadata.confidence_band = 'auto-fill'|'flag-for-review'|'reject'`. The visual treatment is band-aware.
- The audit chip on the drawer reads from `audit_log` filtered by `aggregate_id`. Same pattern as Wave 1.19 audit-log-ui.
- First-paint < 1 s on laptop, < 1.2 s on tablet. The dock view preloads with the location's pending GR list so the operator opens the tablet and is already at the right view.
- Accessibility: the tab strip uses `role="tablist"` + `aria-controls`. Each tab panel is a labeled landmark. Drawer mounts trap focus and restore on close.
