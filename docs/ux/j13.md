# j13 · Cambios retroactivos

**Status: ACCEPTED 2026-05-18.** This document is the spec for the surface
mounted at `/m3/review-queue` (today labelled "Cola revisión" in the top
nav). All 5 §8 open questions were resolved by Master 2026-05-18 03:50 UTC
accepting the v3 roundtable agent recommendations verbatim. See §8 for the
decisions log + §10 for follow-up spec edits identified by the agent.

---

## 1. What this surface is, in one sentence

**A reviewer's worklist of upstream data changes that downstream artefacts
inherited but a human has not yet acknowledged.**

Not to be confused with j12 (Foto-ingestión review): j12 is the *primary*
HITL — "the AI just extracted this, sign it before it lands." j13 is the
*retroactive* HITL — "this thing was already signed, but its source data
just moved; do you still stand behind the original sign-off?"

The two surfaces serve different jobs and *should not* live in the same
nav slot. They should be discoverable from different starting points (j12
from the daily ops cluster, j13 from a "things that need your attention"
badge on the relevant downstream artefacts).

## 2. Personas + JTBD

| Persona | Job |
|---|---|
| Owner Roberto / Iker (§1.1) | "Tell me if anything I signed last week is now stale, and let me re-affirm or correct in <10 s per item." |
| Manager / supervisor | "Triage the queue so my staff isn't acting on outdated cost sheets / recipe sheets / label data." |
| EU AI Act compliance officer | "Prove that human oversight exists for *every* AI-touched artefact, not just at first sign-off (Art. 14 §4(c) ongoing oversight)." |

## 3. When does an item land here?

A row appears in `m3_review_queue` when an upstream change *would have*
changed a downstream artefact that was previously human-signed, but we
deliberately do not auto-propagate. Examples:

| Upstream change | Downstream artefact | Why it queues |
|---|---|---|
| Photo-ingestion extracted a new unit price for a SKU | Recipes referencing that SKU now have different cost | Owner signed the recipe; we don't silently change cost without re-affirmation |
| Recipe edit to allergens override | Labels printed for menu items containing that recipe | Article 21 allergen claim is legally binding; print-time signature must re-issue |
| Supplier cert lapse | Active POs with that supplier | Procurement must acknowledge before next receive |
| Lot-level downgrade after photo review | All menu items historically consuming that lot | Forward-trace bookkeeping |

The queue is opinion-free: it captures *every* deferred propagation event,
sorted by impact + recency. The reviewer decides per row.

### Header
- H1: "Cambios retroactivos" (Fraunces 600 per A1).
- Eyebrow: "Cambios upstream que afectan firmas existentes."
- Filter dropdown (drawer on mobile): by category (coste, alérgenos,
  procurement, lot), by signer (me / equipo / todos), by date range.

### Tabs
- **Pendiente**: default, sorted by impact desc (€ delta * volume).
- **Resuelto**: last 30 days of acknowledged/re-signed items.
- **Todo**: chronological log for audit-trail browsing.

### Row anatomy
- Severity dot — paprika if impact >5% cost delta or allergen-relevant;
  amber if 1-5%; mute otherwise.
- Headline (1 line): subject + delta summary.
- Body (2 lines): firmado <when> by <who>; cambio detectado <when> por <event>.
- Two primary actions inline:
  - **Re-firmar con nuevo X** (confirm-and-propagate). Becomes the
    new audit_log signature.
  - **Mantener firma original** (acknowledge-and-discard). Writes an
    audit_log "operator declined retroactive propagation" event.
- Tertiary: **Ver diff →** opens a side-panel with before/after of the
  affected artefact (cost recalc, allergen matrix diff, etc.).

### Empty state
- Per A5: `<EmptyStateCard Icon={CheckCircle2} title="Sin cambios retroactivos pendientes" body="Todas las firmas están al día con sus datos fuente." />`

## 5. IA placement (where does this surface live)

**Not in top-nav as a primary tab.** Per the v2 roundtable consensus,
the surface should be:

1. **Badge-driven**: a count badge on relevant downstream surfaces
   (`Recetas → 3 pendientes`, `Etiquetas → 1 pendiente`,
   `HACCP → 2 pendientes`) that links here pre-filtered.
2. **Always-on shortcut**: a small "Cambios retroactivos (12)" pill in
   the Dashboard header for the Owner persona — visible only when count
   > 0.
3. **Direct URL preserved**: `/m3/review-queue` stays accessible for
   bookmarks + audit + integrations.

The current top-nav "Cola revisión" entry should be **removed** in a
follow-up slice once the badge pattern lands.

## 6. Acceptance criteria

- [ ] H1 renders in Fraunces 600 per A1.
- [ ] Default tab "Pendiente" loads in <500 ms for orgs with ≤500 queued
      items.
- [ ] Each row offers exactly 2 primary actions (no decision fatigue
      from "Other options" dropdowns).
- [ ] Re-firmar writes an `*_RETROACTIVE_RECONCILED` audit_log event
      with reference to the original signer's event_id.
- [ ] Mantener firma writes an `*_RETROACTIVE_DECLINED` event with
      free-text reason (200 char limit).
- [ ] Diff side-panel renders before/after JSON via the existing
      `<CorrectionsHistoryDiffModal>` primitive (no new diff component).
- [ ] Empty-state uses EmptyStateCard with secondary CTA
      "Ver con datos de ejemplo" (depends on B-3 demo seed).
- [ ] Mobile: row collapses to 2-line summary; tap expands inline.
- [ ] Keyboard: `j/k` to navigate rows, `r` to re-sign, `m` to maintain,
      `d` to open diff (per shared shortcut table in DESIGN.md §7).
- [ ] EU AI Act Art. 14 §4(c) compliance: every action is logged with
      actor + timestamp + before/after state (the diff itself becomes
      the audit_log row).

---

## 7. What is explicitly NOT in scope

- Auto-propagation toggles (would defeat the purpose — this surface
  exists because we *chose* not to auto-propagate).
- Bulk re-sign (operations on >1 row at once). Each row is a deliberate
  legal/operational re-affirmation; batching dilutes that.
- Notification preferences. Lives in `Configuración → Avanzado` per
  general settings architecture.

## 8. Decisions log (2026-05-18 — all questions RESOLVED)

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | **Naming** | **"Cambios retroactivos"** | v3 agent unanimous; avoids j12 collision; non-jargon for Owner persona |
| 2 | **Top-nav removal timing** | **After badges land on Recetas+Etiquetas+HACCP** | v3 agent 4/5; avoids "where did the queue go" hole during the migration window |
| 3 | **Mantener firma confirm pattern** | **Tiered: 1-click with auto-stamped "non-material" reason below impact threshold; typed reason (200 char) above** | v3 agent 4/5; compliance accepts if non-material stamp is audit-machine-readable; threshold TBD (likely €5/item or 1% of dish margin) |
| 4 | **Re-sign default state** | **Default-new** (re-sign button defaults to new value) | v3 agent 4/5; diff side-panel makes the change explicit pre-submit, so accidental drift is mitigated by the affordance not the default |
| 5 | **Dashboard pill visibility** | **Always-on with `0 / N esta semana` zero-state, venue-scoped** | v3 agent unanimous; teaches the affordance + lets ops watch the queue depth trend over time |

All 5 decisions are Master-approved as of 2026-05-18 03:50 UTC, accepting the

## 10. Spec follow-up edits (identified by v3 agent — TODO before code lands)

The v3 roundtable agent flagged 6 spec edits to incorporate before
implementation begins. They're not breaking; track here:

1. **§3 model drift** — clarify that the table is *not exhaustive* of all
   possible upstream→downstream propagations; future M4 / OFF mirror /
   external feeds will add rows.
2. **§4.2 row variant matrix** — formalise the 3 row variants (impact <1% /
   1-5% / >5%) with explicit severity dot + confirm-pattern (per §8 #3
   decision).
3. **§4 severity primitive reference** — bind the severity dot to a single
   reused primitive (`<SeverityDot level />` per DESIGN.md §6 colour scale,
   not a new ad-hoc component).
4. **§5.2 always-on Dashboard pill** — already decided in §8 #5; spec out
   the `0 / N esta semana` zero-state phrasing + venue chip placement
   relative to the pill.
5. **§6 impact-threshold co-sign** — when impact >5% AND the affected
   artefact has multiple signers (e.g. Owner + Manager both signed the
   recipe), require both to ack OR escalate to Owner only. Compliance
   asks for this; needs a Master pick.
6. **§7 cluster-confirm clarification** — "no bulk re-sign" stays, but
   *cluster-confirm of related items* (e.g. all rows triggered by the
   same upstream event) is allowed via a single typed-reason apply-to-all.
   Disambiguate from generic bulk.

Item #5 is the only one that introduces a NEW open question (multi-signer
co-sign vs Owner-only escalation). Park as `j13 open question #6` for
Master before §6 §C-1 implementation.
