# J4 — Cycle detection on save (edge case)

A sub-recipe can reference other recipes, which can reference others. If a save creates a loop (`A → B → A` or any cycle in the graph), the recipe must not be saved. The error UX names both endpoints and the direction so the chef can resolve it without reading documentation.

## Goal

Block any save that introduces a sub-recipe cycle, and tell the chef which references created the loop.

## Trigger

Lourdes edits `Salsa de tomate` and tries to add `Salsa madre` as a sub-recipe. But `Salsa madre` already references `Salsa de tomate` as one of *its* sub-recipes. Saving would close the loop.

## Walkthrough

1. Lourdes opens `Salsa de tomate` and taps **+ Sub-recipe**. `RecipePicker` autocompletes `Salsa madre`. She taps it.
2. The sub-recipe row appears in the table. The local UI does not yet detect the cycle — composability is checked on **save**.
3. Lourdes taps `Save recipe`. The backend runs the cycle-detection graph walk (depth-first, capped at depth 10 per the M2 NFR).
4. A cycle is found: `Salsa de tomate → Salsa madre → Salsa de tomate`.
5. The save is rejected. An inline error banner renders at the top of the editor pane (above the title), the new sub-recipe row is highlighted with a `--destructive` left edge, and the primary CTA `Save recipe` enters its disabled state until the offending row is removed or the loop is broken.

## The error UX

The banner names **both endpoints** and **the direction** of the loop. The text is matter-of-fact, kitchen-professional (DESIGN.md §7 Copy):

> ⚠ **Cannot save: this would create a loop.**
>
> `Salsa de tomate` → `Salsa madre` → `Salsa de tomate`
>
> Remove one of these sub-recipe references to save.

Visual rules (per DESIGN.md):

- Banner background: `--warn-bg` (`#F2E3C8`).
- Banner border: 1.5 px `--destructive` (`#A8392F`) on the left edge only.
- Icon: geometric SVG triangle (not emoji), `--destructive` fill.
- Text: `--ink` body, with the recipe names rendered in the recipe-title family (`--font-serif`) for instant recognisability. The arrow `→` is `--mute`.
- The offending row in the ingredient table gets a 2 px `--destructive` left edge stripe (this is the **only** place a side-stripe is allowed — the J4 error state — and it is documented as an exception to DESIGN.md §6 "no side-tab" rule).

## Components used

- Inline error banner (a J4-specific pattern; not a reusable component until repeated elsewhere).
- `RecipePicker` (the source of the offending pick).
- `Save recipe` button in `disabled` state (DESIGN.md §4 universal states).
- Offending row indicator (the `--destructive` left edge — exception to §6).

## Capabilities satisfied

FR6 (cycle detection on save).

## Decisions specific to J4

- **Detection on save, not on pick.** Local UI does not pre-check cycles on every keystroke — it would be wasted work for a rare condition, and the backend already does the authoritative check. Tradeoff: the chef sees the error a beat later than ideal, but the path is honest about where the truth lives.
- **Both endpoints + direction in the message.** "Cycle detected" alone is useless. The chef needs to know exactly which references to remove. The arrow notation reads in 2 seconds.
- **Side-stripe is the deliberate exception.** DESIGN.md §6 forbids `border-accent-on-rounded` / side-tab patterns generally. J4 is the documented exception because the chef must locate the offending row instantly among many — and a left-edge stripe on the row is the highest-signal locator we have without inventing new chrome.
- **Depth cap at 10.** Per the M2 NFR — no recipe will ever realistically nest deeper. If the graph walk reaches depth 10 without resolving, the save is rejected with a different message ("recipe nesting too deep, simplify by inlining a sub-recipe"). That's a separate edge state, not part of J4.

## Notes for implementation

- The cycle-detection backend is documented in the M2 architecture ADR (see [architecture-decisions.md](../architecture-decisions.md)) and is invoked synchronously during the save transaction. Round-trip latency budget: < 500 ms for the 95th percentile (typical recipe graph is shallow).
- Screen-reader output for the banner: "Cannot save. This would create a loop. Salsa de tomate, then Salsa madre, then Salsa de tomate. Remove one of these sub-recipe references to save."
- Once the offending row is removed (or the user discards the change), the banner clears with a 300 ms `--accent`-tinged fade-out. Any other resolution path (renaming, etc.) does not auto-clear; the next save attempt re-checks.
