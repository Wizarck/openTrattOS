# J5 — WhatsApp recipe creation (forward-looking, M2.x)

This journey is **not** in M2 MVP scope. It is recorded so that M2 architecture and design choices do not preclude it. M2.x will revisit with its own PRD, full design, and mock. Treat this doc as a contract M2 must keep open.

## Scenario

Sunday evening, Lourdes is making risotto for her family. She thinks "this would be a good winter special." She doesn't want to open a tablet. She opens WhatsApp on her phone, sends Hermes a photo of the dish + the message:

> *"Hagamos un risotto de calabaza para el menú de invierno. La calabaza la asé a 200 °C 30 min antes de añadirla."*

Hermes (the agent) replies in WhatsApp, walking her through composing the recipe via chat:

- *"Perfecto. ¿Para cuántas raciones?"*
- *"¿Qué arroz usaste — Carnaroli, Arborio?"*
- *"Por la foto detecto: calabaza, arroz, caldo, parmesano, mantequilla, cebolla. ¿Falta algo?"*

After ~6 messages and ~3 minutes, Hermes has enough to draft a **`status: pending_review`** Recipe in openTrattOS. It does not auto-publish. It lands in an **inbox** that Lourdes reviews from her tablet on Monday morning when she's back in the kitchen.

## Goal (M2.x)

Let trusted personas (Manager+, allowlisted) create draft recipes through WhatsApp (or any allowlisted channel via the same MCP contract) without leaving the device they happen to have at the moment of inspiration. Drafts are reviewed in-app before publishing — never auto-published.

## What M2 must keep open (the contract)

This is the load-bearing part of this doc. M2 implementation must not foreclose:

1. **Recipe lifecycle includes a `pending_review` (or `draft_external`) state.** Beyond the M2 MVP states (draft / published / archived), the architecture must allow a fourth that represents "draft created by an agent on behalf of a human user, awaiting human review before publishing". The state machine ADR captures this. Without it, we will be retrofitting the entire publish flow in M2.x.
2. **Every Recipe row carries `created_via_channel`.** A nullable string column or jsonb metadata field that records which channel produced the draft (`web`, `whatsapp`, `telegram`, `cli`, `agent`). Used by the inbox to render the channel chip and by audit to reconstruct provenance.
3. **The MCP server `opentrattos-recipes` is the single contract for both UI and agents.** Per [ADR-013](../architecture-decisions.md), the human UI and Hermes consume the same API. M2 implements the human UI; M2.x adds the agent path on top. No special agent endpoints.
4. **Phone number is on the User entity.** Per the User retrofit in the M2 data model (E.164 string, nullable), so the routing layer can match an inbound WhatsApp message to a User without a separate identity table.
5. **Allergen + EU 1169/2011 compliance is non-negotiable on the agent path too.** Hermes-drafted recipes must pass the same allergen / Article 21 emphasis rules before publishing. The M2.x review surface enforces this — Lourdes cannot publish a draft if its `AllergenBadge` is missing required content.
6. **Multimodal input is supported via the MCP layer.** WhatsApp sends images. Hermes uses a vision model (small Gemma via OpenRouter, per the M2 ADR on AI yield-suggestion model — same key wallet) to extract candidate ingredients from the photo. The UI surface is unchanged; the work happens in the agent layer before drafting.

## Design intent (visual sketch — to be drafted in M2.x)

- **WhatsApp side**: standard chat. No openTrattOS chrome. Voice notes accepted in M2.x v2 (transcribed via the same vision/text model stack).
- **Inbox surface in openTrattOS** (tablet): a sidesheet or top-nav badge ("3 drafts pending review"). Each draft renders as a card with: dish name, channel chip (`WhatsApp` in `--mute` text on `--surface`), date, and a `Review draft` CTA. Tapping opens the J1 recipe edit screen pre-populated, with a banner: *"This draft was started in WhatsApp by Lourdes on Sunday at 19:42. Review the AI-prefilled values before publishing."*
- **Channel trace**: every draft view has a small `audit` link in `--mute` that opens the original WhatsApp conversation thread (read-only). Provenance must be recoverable.

## Components needed (M2.x — not in M2)

- `InboxItem` — the draft card in the inbox sidesheet.
- `AgentDraftReviewBanner` — the banner on top of the J1 edit screen when entering review mode.
- `ChannelTrace` — the audit-link pattern that opens the source conversation.

These are tracked here so the components catalogue ([components.md](components.md)) records them as `status: deferred-M2.x` rather than appearing as gaps.

## Capabilities (M2.x)

FR41–FR45 (channel routing, multimodal input, draft lifecycle, review surface, allergen enforcement).

## Open questions for M2.x PRD

- **Allowlist policy.** Which roles can create drafts via WhatsApp? Initial proposal: Manager + Owner only. Staff (Line Cook) deferred to a later iteration. Owner-level toggle to enable / disable per organisation.
- **Inbound WhatsApp routing.** How are messages from numbers not in the User table handled? Reject silently? Reply with onboarding link? Initial: reject silently and log; revisit if the support load justifies onboarding via WhatsApp.
- **Voice notes.** v1 (M2.x): photo + text only. v2: voice-note transcription. Defer the v2 question until v1 has a quarter of usage signal.
- **Cost / quota.** Vision-model calls cost money. Per-org budget cap with alerting. Defaults TBD with the M2 AI-yield-model wallet.

## Why this doc exists

If we don't write down the M2.x contract during M2, we will design ourselves into a corner. The state machine, the channel column, and the MCP single-contract rule are exactly the kind of decisions that are cheap to honour now and expensive to retrofit.

This doc is the contract. M2.x picks it up.
