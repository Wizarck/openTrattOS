# J10 — Head Chef logs a CCP reading on the tablet (HACCP path)

The recurring HACCP capture surface. Carmen logs the cooling curve breakpoint at the end of service; Mikel logs the morning cooler temp via Hermes (WhatsApp / Telegram / chat widget — different surface, same MCP capability `haccp.record-ccp-reading`). This doc anchors the web/tablet rendering — the agent-side rendering is the same data via `Hermes`. The visual surface is [mock-j10-haccp-record.html](variants/mock-j10-haccp-record.html).

## Goal

In ≤30 seconds, Carmen (i) picks the CCP she is recording against, (ii) enters the reading, (iii) reads back the system's spec range to confirm in-spec or trigger corrective action, and (iv) signs the row to audit_log. The flow is glance-driven; oily-finger friendly; resilient to a 5-second pause when she's interrupted by service.

## Trigger

End of service, ~15:30. Carmen wipes her hands, taps the tablet at her station. The home screen presents `Registrar lectura HACCP` as a primary CTA among the day's CCPs — those overdue carry a warning eyebrow. Tap → this form.

## Walkthrough

The numbers below correspond to the regions of [mock-j10-haccp-record.html](variants/mock-j10-haccp-record.html).

1. **Header.** Eyebrow: `HACCP · Lectura de PCC`. Headline: `Cooling curve · cámara entrante`. The CCP name carries a one-word Fraunces wink (the canonical PCC label, per DESIGN.md §3 one-serif-wink rule). Below, a `--mute` eyebrow: `Carmen Aldai · Casa Aitona Bilbao · 15:32 CEST · referencia FSMS-2026-v2`. The FSMS-standard reference links to the spec sheet — Carmen taps if she needs to remember the exact threshold.
2. **CCP picker (collapsed by default).** If Carmen arrived from a CCP-specific deep-link the picker is collapsed to a single line: `Cooling curve · cámara entrante · cambiar →`. If she arrived from the generic "log a reading" entry, the picker opens with the day's CCPs listed: each row carries name + last reading + due-by countdown. Selection collapses the picker.
3. **Reading input.** A single primary input. Type depends on the CCP definition (per FSMS standard config): numeric (°C, %, pH, time-elapsed), checkbox (visual inspection: clean/not-clean), or multi-select (allergen contamination check). The mock shows the numeric case (cooling curve final temp). Input: 60 px tall, `--ink` text on `--surface`, tabular-nums, decimal-aware keyboard on touch devices. Placeholder: `°C — entre -2 y 2 °C`.
4. **Spec range readback (live).** As Carmen types, a `--mute` line below the input shows the in-spec range from the FSMS standard: `Rango aceptable: -2 a 2 °C`. When the typed value falls in-spec the line turns `--success` with a `✓ Dentro de rango`. When out-of-spec it turns `--destructive` with `⚠ Fuera de rango · se requiere acción correctiva`. The readback is the system's contract; no surprises, no hidden validations.
5. **Corrective-action linkage (conditional).** If the reading is out-of-spec, a section mounts below: `Acción correctiva requerida (FR12)`. A dropdown picker of pre-defined correctives (per FSMS-standard config + free-form). Once selected, a small text area for notes (`opcional`). Above the dropdown a `--mute` reference to the M2 incident pattern: `Esto crea una entrada vinculada en audit_log con tu lectura como contexto`.
6. **Confirm row.** Two actions: `Cancelar` (ghost, left) and `Firmar lectura` (primary `--accent`, right). The primary CTA disabled until the reading input is non-empty (and the corrective-action picker filled if out-of-spec). Touch target 64 px tall — oily-finger friendly.
7. **Confirmation interstitial (inline).** Tap → strip mounts in place of the form: `✓ Lectura firmada · audit_log AL-2026-189587 · 15:32:14 CEST · firma Carmen Aldai`. Two ghost actions: `Registrar otra lectura` and `Volver al panel`. Auto-redirect to panel after 8 seconds if no action taken.
8. **Recent readings strip (right sidebar on landscape tablet).** Last 5 readings for this CCP, oldest at bottom. Each row: timestamp + value + actor + in/out-of-spec glyph. Carmen scans the strip in 1 second to confirm her reading "feels right" against recent history — surfacing pattern (e.g. a slow drift over 4 days).
9. **Out-of-spec sticky warning (if last reading was out).** If the most recent prior reading was out-of-spec AND no corrective-action linkage exists yet, a sticky `--destructive` strip mounts at the top of the surface: `Lectura previa fuera de rango sin acción correctiva · revisar antes de firmar nueva lectura`. Carmen sees this immediately on landing; she cannot ignore it.

## Components used

`CcpPicker` · `ReadingInput` (numeric / checkbox / multi-select variants) · `SpecRangeReadback` · `CorrectiveActionPicker` · `RecentReadingsStrip` · `OutOfSpecStickyWarning`. See [components.md](components.md). The eyebrow + headline + breadcrumb reuse `AppLayout` from M1+M2.

## Capabilities satisfied

FR9 (CCP capture per FSMS standard config), FR10 (corrective-action linkage when out-of-spec), FR11 (FSMS-standard reference inline), FR12 (in-spec validation + readback), FR13 (audit_log envelope per reading). Cross-reference [PRD-M3 §FR Catalogue HACCP Record System](../_bmad-output/planning-artifacts/prd-m3.md). The same capability is invoked by `Hermes` for Mikel's staff-role logging via WhatsApp — identical contract, different surface (no RBAC-differentiated UX for the basic act of logging, per PRD-M3 §Journey 6).

## Edge cases

- **Carmen interrupted mid-entry.** Input persists locally for 10 minutes. If she returns within the window, the value she had typed is still there, with a mute eyebrow `Borrador desde hace 4 min · ¿continuar?`. After 10 minutes the draft is discarded — the system never silently signs a stale draft.
- **FSMS-standard config changes between draft and submission.** Spec range is locked to the version active at submission time. If a new FSMS version became effective during the draft, the readback surfaces a mute warning `FSMS-2026-v2 → v3 vigente · revise el rango`. Carmen can pick the version explicitly via a ghost link.
- **Reading device offline.** The input + submit work offline; the audit_log envelope queues locally and writes on next connectivity. The confirmation strip reads `Lectura encolada · se firmará al recuperar conexión` until the queue flushes.
- **Out-of-spec without corrective-action picker available** (FSMS standard misconfigured). The form fallback is a free-form text area; the CCP is still recorded with the action note. A mute eyebrow surfaces `Standard sin correctivas definidas · contacte al Owner para revisar config`.
- **Multiple staff logging same CCP simultaneously.** Each reading is its own audit_log row; the RecentReadingsStrip refreshes on confirmation and shows both. No conflict resolution — the truth is the chronology.
- **Carmen taps Cancel after typing.** Confirmation modal-strip: `Descartar lectura sin firmar?`. Confirm discards; cancel returns to the form with the value preserved. (Single-step confirm — accidental cancel is the failure mode to protect against, not deliberate cancel.)
- **The CCP has not been read in 48 hours.** A mute eyebrow on the headline: `Última lectura hace 51h · vencido el martes 15:30`. Carmen sees overdue context without it blocking the new reading.
- **The same operator submits two readings within 1 minute.** Both accepted; the chronology shows both. The second reading carries a mute eyebrow `Lectura adicional 32 s después de la anterior` — surfaces possible double-tap. The operator can soft-delete the duplicate via the RecentReadingsStrip (ADR-009 soft-delete inheritance).

## Decisions specific to J10

- **Spec range readback is LIVE, not on submit.** The persona's mistake mode is "typed wrong number, didn't notice". Live readback prevents the round-trip and catches the typo before the audit_log row is written. (DESIGN.md §1 principle 8 — boring before clever; the live affordance is conventional and load-bearing.)
- **Out-of-spec MUST trigger corrective-action picker BEFORE submission.** Allowing a free-form out-of-spec submission decouples the reading from the response — exactly the gap that recall investigations later uncover. The architecture (ADR-031 audit_log indexing) treats CCP-without-corrective as a queryable lapse; the UX rule prevents the lapse from being created in the first place.
- **No "save draft" affordance.** A draft is an unsigned reading — the regulator would ask "was this reading reviewed before signing?". The local 10-minute persistence is a kindness for interruption recovery, not a workflow primitive. Drafts never enter audit_log.
- **CCP picker collapses on selection.** Once the CCP is chosen, the picker is no longer load-bearing on the surface. Collapsing it removes a column from the eye's scan and gives the input + readback more room.
- **Recent readings strip is read-only.** Carmen can NOT correct a past reading from this strip; corrections happen via a separate amendment flow that produces its own audit_log row. (Trust principle: existing readings are immutable; new readings can re-state.)
- **Sticky warning at top for prior-out-of-spec-without-action.** Out-of-spec readings without corrective linkage are the most consequential gap; surfacing them before Carmen logs a new reading interrupts a possible repeat-pattern (the kitchen is operating with a known hazard).
- **Identical contract across web and Hermes.** Mikel via WhatsApp hits `haccp.record-ccp-reading` with the same envelope; the readback he receives is a Hermes text rendering of the same spec range. The web mock is one rendering of the canonical contract; the contract is API-first per ADR-002.
- **Touch target 64 px on primary CTA, 56 px on input.** Above the 48 px standard because the operator may have wet/oily hands. (DESIGN.md §5 — tablet-first kitchen-real, principle 3 inherited.)
- **The eyebrow includes FSMS-standard reference.** Carmen's mental model is "I'm logging against THIS standard". Surfacing the version inline (with a tap to open the spec sheet) honors that mental model without forcing a separate navigation.

## Notes for implementation

- The input type per CCP is derived from the FSMS standard config (`fsms_standards.input_type` enum: numeric / checkbox / multi-select / range). The component selects the rendering variant.
- Live readback runs client-side; the spec range is loaded with the CCP picker selection and stays in component state. No server round-trip per keystroke.
- Local draft persistence uses IndexedDB scoped to `(organization_id, ccp_id, actor_id)`. The 10-minute TTL is a client-side timer; on submit the draft is cleared.
- Submission writes a `haccp.record-ccp-reading` envelope per Wave 1.19 audit-log canonical (ADR-025). The `payload_after` includes: `reading_value`, `reading_unit`, `spec_min`, `spec_max`, `in_spec` (boolean), `corrective_action_id` (nullable), `fsms_standard_version`.
- The RecentReadingsStrip reads from `audit_log` filtered by `(aggregate_type='haccp_record', aggregate_id LIKE ccp_id_prefix)` — same query pattern Wave 1.19 audit-log-ui uses.
- Offline queue uses the existing service-worker pattern from M1+M2 (Wave 1.5 offline ingredient picker). Reuse; do not reinvent.
- First-paint < 800 ms on slow Wi-Fi. The CCP picker + last 5 readings preload with the home dashboard so the surface is warm.
- Accessibility: live readback uses `aria-live="polite"` so the in-spec/out-of-spec status announces to screen readers. The corrective-action picker mounts with focus on the dropdown for keyboard flow.
- The Hermes equivalent: `Hermes.invoke('haccp.record-ccp-reading', { ccp_id, value, unit, actor_user_id })` returns the same readback as text — Carmen's web flow and Mikel's WhatsApp flow are two surfaces of one capability.
