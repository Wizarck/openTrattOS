# J2 — Head Chef investigates a cost spike (audit path)

The audit deviation from [J1](j1.md). Where J1 builds, J2 explains. Lourdes lands on this surface only when something has moved against her — the live cost on a published recipe is no longer the live cost she signed off. The visual surface is [mock-j2-cost-spike.html](variants/mock-j2-cost-spike.html); this doc anchors that mock to capabilities, components, and decisions.

## Goal

In a 2-second glance Lourdes identifies which component drove the cost spike on `Tagliatelle Bolognesa` over the last 7 days, sees how much margin has been eroded, and acts — switch supplier, reformulate, or raise the sale price — without leaving the audit surface.

## Trigger

Tuesday afternoon, between services. Lourdes opens the recipe and notices the live cost reads `€4.12 / kg`, up from `€3.45 / kg` last week. She taps the **What changed?** affordance on the recipe summary strip. The audit surface opens.

## Walkthrough

The numbers below correspond to the regions of [mock-j2-cost-spike.html](variants/mock-j2-cost-spike.html).

1. **Header.** Title reads `Tagliatelle Bolognesa — Cost audit`. The recipe name carries the Fraunces serif (the same one-serif-wink rule as J1, see DESIGN.md §3); `— Cost audit` stays system sans so the surface is clearly an audit context, not a new editor. Eyebrow: `Audit · 7-day window`.
2. **Comparison-window chip group.** Three chips: `vs 7 days ago` (selected), `vs 30 days ago`, `vs 90 days ago`. Selected chip carries `--accent` border + `--ink` text; unselected chips carry `--border` + `--mute` text. Tapping a chip re-renders the table (FR13 — configurable window). Default is 7-day because recency is the recurring kitchen question ("what changed since last week?"); 30-day and 90-day exist for renegotiation cycles and seasonality.
3. **Sticky action bar.** Three actions, left-to-right: `Back to recipe` (ghost), `Switch supplier` (secondary, disabled until a row is selected), `Save & re-publish margin` (primary `--accent`). The bar is sticky so the chef can act from anywhere on the table without scrolling. Touch targets ≥ 48 px (DESIGN.md §5, §7).
4. **`CostDeltaTable` — the main surface.** No card around it (DESIGN.md §6 — the table is not in a card). Columns: Component / Source / Qty / Cost (was) / Cost (now) / Δ / Δ %. Rows sorted by absolute delta descending (FR15). The Ground beef row sits at the top with `Δ +€0.58 (+35.8 %)` rendered in `--destructive` (cost going up = bad for the chef). Hover on `pointer:fine` reveals an inline `Switch supplier →` action on the spike row; on `pointer:coarse` the same action is reachable via the sticky `Switch supplier` button after the row is tapped to select.
5. **No-change rows are de-emphasised.** `Salsa de tomate`, `Onion`, `Celery`, `Red wine` have their delta cells in `--mute` with the `—` glyph instead of `+€0.00`. The eye skips them on the first glance, which is exactly what the persona needs in 2 seconds. (Decision documented below.)
6. **MarginPanel (right sidebar).** Single bordered panel (DESIGN.md §6 — flat full-bordered, no nested cards). Shows `Cost €3.45 → €4.12` with the new value in `--ink` and the old in `--mute` strikethrough; `Margin 80.8 % → 77.1 %` with the same pattern; `Target 70.0 % ✓` in `--success` (still met but tighter). The accent never appears here — this is a margin domain, not a brand-CTA domain (DESIGN.md §4 `MarginPanel`).
7. **Aggregate strip (bottom).** `Total cost +€0.67 / kg (+19.4 %)` rendered with `font-variant-numeric: tabular-nums lining-nums` so digits don't dance when the chip toggle re-renders the row.
8. **Drill-into supplier override.** From the spike row, tapping `Switch supplier →` opens `SourceOverridePicker` inline on the source cell — the M2 `SupplierItems` list sorted preferred → cheapest. Clicking out commits. The audit surface stays mounted; Lourdes never loses her place. (FR15 — attribution to a specific `SupplierItem`.)
9. **Re-publish.** Once the supplier is changed (or sale price raised), the primary CTA `Save & re-publish margin` lights `--accent`. Tap → MenuItem updates; cost rollup recomputes in < 200 ms (NFR); the audit surface refreshes the deltas against the new baseline.

## Components used

`CostDeltaTable` · `MarginPanel` · `SourceOverridePicker` (drill-in). The chip group is a local control, not a named component. See DESIGN.md §4 for tokens + states.

## Capabilities satisfied

FR13 (per-component cost-history with configurable window), FR14 (system recomputes dependent Recipe costs when SupplierItem prices change — happens upstream of this surface), FR15 (per-component delta with attribution). Cross-reference [PRD-M2 §FR catalogue](../prd-module-2-recipes.md).

## Edge cases

- **All deltas zero.** The chef toggled to a window where nothing moved. The table renders with every row in the no-change visual treatment; the aggregate strip reads `Total cost +€0.00 (no change)`. The `Switch supplier` action stays disabled. (Avoids a confusing empty state — the data is the data.)
- **Single component spiked, others all dropped.** The spike still sorts to the top (absolute delta wins), but the dropping rows render in `--success`. The aggregate may be net-positive or net-negative; the strip carries the sign.
- **Component removed from recipe between the two timestamps.** Renders as a row with `—` in the new-cost column and a `removed` mute eyebrow. Δ is the negative of the old cost. (Honest about the change; doesn't silently drop history.)
- **Component added between the two timestamps.** Mirror — `—` in the old column, `added` mute eyebrow, Δ is the new cost.
- **Sub-recipe rolled up cost moved.** The sub-recipe row shows the rolled-up delta; tapping the row drills into the sub-recipe's own audit (`Salsa de tomate — Cost audit`), recursively. (FR15 attribution flows through composition.)
- **Stale supplier price.** If the supplier hasn't reported in N days, the source cell carries a mute `stale 14d` eyebrow. The delta still computes against last-known price; the chef has the context to decide whether to trust it.

## Decisions specific to J2

- **Sort by absolute delta descending, not alphabetical, not by current cost.** The persona is asking "what changed?", not "what does this recipe contain?" or "what is most expensive?". Sorting by absolute delta puts the answer on row 1, every time. Alphabetical sort would force the chef to scan all seven rows to find the spike, which costs ~5 seconds — over budget for the 2-second-glance NFR. Sorting by current cost would surface ground beef every week regardless of whether it moved, and would bury a 35 % spike on a small-line-item ingredient.
- **Default window: 7 days.** The recurring kitchen question is "what changed since last week's menu meeting?". 30-day and 90-day chips exist for renegotiation cycles and seasonality but are not the default — defaults serve the recurring case (DESIGN.md §1, principle 8 "Boring before clever").
- **Margin compression shown alongside the cost delta.** The chef doesn't only care that ground beef rose 35.8 %; she cares whether her margin still hits target. The MarginPanel sits in the right sidebar so the cost story (left) and the margin story (right) read in one glance. Putting the margin in a footer or a separate tab would force a second glance — over budget.
- **No-change rows de-emphasised, not hidden.** Hiding them would make the table feel sparse and would also make "did this row not change?" indistinguishable from "is this row missing?". Muting the delta cell + using `—` keeps the row readable but visually quiet — the chef's eye skips it on the first glance and can return to it on a second.
- **Negative-delta colour rule: `--destructive` on cost-going-up.** The sign of the number and the colour are independent. A positive delta (cost rose by €0.58) is `--destructive` because cost-going-up is bad for the chef; a negative delta (cost fell) is `--success`. This is counter-intuitive if you read "+ = green" from finance dashboards, which is exactly why the rule is documented here. (DESIGN.md §2 — semantic colour, not decorative.)
- **Drill into `SourceOverridePicker` from the spike row.** The audit surface is the surface where Lourdes acts. Forcing her back to the recipe edit screen to switch supplier would break the 30-second-audit budget (PRD §M2 NFR). The picker mounts inline on the source cell; click-out commits.
- **`Switch supplier` reachable on `pointer:coarse` even though the inline reveal is hover-only.** The kitchen tablet is `pointer:coarse`. Hover-only affordances on this device are bugs (DESIGN.md §4 universal states, "hover is never the only affordance"). The sticky action bar carries the same affordance, surfaced when the row is selected via tap.

## Notes for implementation

- The cost-history backend exposes per-`SupplierItem` price snapshots; the audit query is "give me the recipe's effective component costs at timestamp T1 and T2, attributed to the SupplierItem that was active at each point". FR14 keeps the rollup honest as upstream prices change; FR13 is the read query for the window.
- The chip group control re-renders the table client-side from a pre-loaded 90-day window so the toggle is instant — no spinner, no layout shift. Slow Wi-Fi only matters on first paint.
- The "stale supplier price" eyebrow is a nicety; the data shape supports it but it's not on the M2 critical path. Ship without it if it pushes the milestone.
- `MarginPanel` is shared with [J1](j1.md). The compression visual treatment (old → new with strikethrough on old) is J2-specific; in J1 there is only one value per row. Document this divergence in `components.md` so a future agent doesn't unify them prematurely.
- The audit surface is read-only by default. The two write actions (`Switch supplier`, `Save & re-publish margin`) require explicit affirmative action; the chef cannot accidentally re-publish by toggling chips.
- First-paint < 1 s on slow Wi-Fi (NFR carries over from J1). The audit query is bounded — seven rows of the recipe + one rolled-up sub-recipe — so the payload is trivial.
