# J7 — Owner reviews and amends the dispatched recall dossier (post-crisis path)

The aftermath surface. [J6](j6.md) generated and dispatched the incident dossier; J7 is where Iker — or the insurer, or the inspector following up — reads what was sent, when, to whom, and (if warranted) appends an immutable addendum. The original dossier is never re-edited; addenda extend the record. The visual surface is [mock-j7-recall-dossier.html](variants/mock-j7-recall-dossier.html); this doc anchors that mock to capabilities, components, and decisions.

## Goal

In ≤2 minutes, Iker (i) confirms each recipient received the dossier, (ii) re-reads the as-dispatched contents, (iii) downloads the PDF if needed, and (iv) appends an addendum for new information (corrective actions taken, lab results, supplier statements) without touching the original.

## Trigger

The receipt strip in [J6](j6.md) links here once the dossier is dispatched (`Dossier dispatched · 02:21 CEST · ver dossier →`). Iker also lands here from email notifications (`Tu dossier para el incidente IR-2026-0007 fue recibido por aseguradora-claims@…`) or from the standard layout's notification centre during business hours.

## Walkthrough

The numbers below correspond to the regions of [mock-j7-recall-dossier.html](variants/mock-j7-recall-dossier.html).

1. **Standard layout returns.** Unlike [J6](j6.md), J7 uses the standard `AppLayout` — top nav present, breadcrumb visible (`Recall · Incidente IR-2026-0007 · Dossier`). The crisis surface has ended; the operator is reading, not reacting. (DESIGN.md §1 principle 1 — context dictates chrome.)
2. **Status header.** Eyebrow: `Incidente IR-2026-0007 · despachado el 13 mayo 2026 02:21 CEST · ventana legal cumplida en 0h 21min`. The legal-window status reads `✓ Dentro de plazo` in `--success` sage if dispatched before the 4-hour deadline; `⚠ Fuera de plazo` in `--destructive` paprika otherwise. This is the regulator's primary question; surface it first.
3. **`DispatchReceiptCard`** (one panel, flat-bordered, no nested cards). Lists every recipient with delivery status: `info-claims@aseguradora.es ✓ entregado 02:21:14`, `inspector.regional@sanidad.eu ✓ entregado 02:21:09`. Failure rows show `⚠ reintentando · intento 2/3` in `--mute` (per NFR-REL-2 exponential backoff). Permanent failure shows `✗ fallo final` with a `Reenviar manualmente` ghost action. The card's last row is `Original audit_log entry: AL-2026-189447 · ver →` linking to [J12](j12.md) (M3 audit-log browse).
4. **`DossierPreview`** (read-only PDF-style preview embedded). Renders the dispatched PDF inline. The preview is read-only by visual treatment: no edit affordances, every block carries a mute `Dispatched 02:21 CEST` watermark in the gutter. Sections in the PDF order: Incident chronology · Lot provenance · Consumption chain · Corrective-steps placeholder · Signed signature block (Owner's actor-snapshot + system signature). The preview is identical byte-for-byte to what was emailed (FR18).
5. **Download / Re-dispatch row.** Two ghost actions side by side: `Descargar PDF` and `Reenviar a destinatarios`. Re-dispatch opens a small inline picker (NOT a modal) of the original recipient list with checkboxes — the operator can re-send to a subset (e.g. only the failed recipient). Every re-dispatch is its own audit_log entry (canonical chain extended).
6. **Addendum strip (sticky bottom).** A single ghost CTA: `Añadir adenda`. Distinct from [J6](j6.md)'s destructive CTA — this is `--accent` border + `--mute` text, signalling secondary deliberative action. Tapping it expands an inline form below the dossier preview: a text area, optional file attachments (lab results PDFs, supplier statement images), and a single confirm pill `Adjuntar al expediente`. Addenda are immutable once attached; they appear above the original dossier in the chronology view (newest first) and carry their own audit_log envelope.
7. **Chronology rail (right sidebar on laptop, bottom drawer on phone).** A vertical strip listing every event on this incident, oldest at top: `02:14 Investigación iniciada`, `02:18 Lote identificado L-2026-0042`, `02:21 86-flag dispatched + dossier sent`, `02:21 Audit_log AL-2026-189447`, `04:35 Adenda 1 — Lab result: Anisakis negativo`, etc. Every row carries actor + UTC timestamp. The rail is built from the audit_log envelope chain (ADR-025 + ADR-032 hash chaining) — single source of truth.
8. **No "edit dossier" action anywhere on the surface.** The original is immutable post-dispatch. Addenda are the only forward action that touches the record. (Decision documented below.)

## Components used

`DispatchReceiptCard` · `DossierPreview` (read-only) · `AddendumComposer` · `IncidentChronologyRail` · `RecipientList` (re-dispatch). The breadcrumb is a standard `AppLayout` component shared with M1+M2 surfaces.

## Capabilities satisfied

FR18 (pre-formatted incident dossier PDF — render the as-dispatched bytes), FR19 (dispatch via email + retry/backoff + manual re-send), FR20 (≤4h response — surface the legal-window check), and the M3 audit-log envelope chain (ADR-025 + ADR-032). Addendum capability is unlisted in PRD-M3 explicitly; covered implicitly by FR40 (general audit_log envelope writes) — call out in next PRD revision.

## Edge cases

- **Re-dispatch to a subset of recipients.** The picker carries a `Marcar todos` ghost and individual checkboxes. Confirm fires the dispatcher; each recipient produces its own audit_log row. (Iker can re-send only to the failed one without spamming the other recipients.)
- **Addendum with no attached file, only text.** Allowed — many real addenda are short factual updates ("Inspector regional visited, no further action requested"). The composer accepts text-only.
- **Addendum with a large attachment.** Files up to 50 MB accepted via the dispatch service. Beyond that, the operator gets a mute hint `Archivos > 50 MB → comparte URL en el texto`. We do not block; we redirect.
- **Recipient delivery is still in retry phase.** The receipt card row carries `⚠ reintentando · intento 2/3 · próximo 02:25:14` with a live countdown to next retry. The operator can `Cancelar reintentos` (ghost action) if they know the recipient is bouncing for legitimate reasons (mailbox quarantined for the night).
- **Operator views this surface long after the incident.** All data persists. The chronology rail can scroll back days/months. The dossier preview always renders the as-dispatched bytes, even years later (immutable storage per ADR-029).
- **PDF preview fails to render.** The fallback is a plain-text rendering of the dossier sections + a `Descargar PDF` CTA. The data is preserved regardless of rendering capability (graceful degradation).
- **Addendum draft saved without confirm.** The composer auto-saves drafts to local storage; if Iker navigates away and returns, the draft is there. Drafts are NOT in audit_log until the operator confirms.

## Decisions specific to J7

- **The original dossier is never re-edited.** The post-dispatch view is read-only. Edits would invite an "I just fixed the typo before the regulator saw it" defence that destroys the chain of custody. Addenda extend the record forward; the original stays exactly as the regulator received it. (Trust principle locked from PRD-M3 §Journey 5 — Marta's adversarial-aware export inherits the same logic.)
- **Legal-window status (≤4 h) is the first thing the operator sees.** The persona's recurring question — and the regulator's first question — is "did you respond within 4 hours?". Surfacing this in the status header makes the answer scannable in 1 second. Burying it in the chronology would force a scan that wastes attention in an audit context. (DESIGN.md §1 principle 8 — defaults serve the recurring question.)
- **Addendum CTA is `--accent` ghost, not `--destructive` primary.** The post-crisis surface should NOT carry a destructive primary. The action is deliberative ("I have something to add"), not urgent. Making the addendum CTA look like [J6](j6.md)'s dispatch CTA would conflate two very different mental modes. (DESIGN.md §2 semantic colour.)
- **Chronology rail is a sidebar (laptop) / bottom drawer (phone).** The rail is reference material — the operator wants it visible while reading the dossier, not in a separate tab. On laptop it lives in the right sidebar; on phone it collapses to a bottom drawer that swipes up. (DESIGN.md §6 — no nested cards; the rail is a flat list, not a card grid.)
- **Re-dispatch is the same handler as initial dispatch.** No separate "resend" service. Every dispatch — original or repeat — writes its own audit_log row through `recall.dispatch-86-flag` MCP capability (or the email-only variant). This keeps the audit story uniform and removes a forking code path. (ADR-039 email dispatch DI inherits this.)
- **`IncidentChronologyRail` reads from audit_log envelopes, not from a derived store.** The rail is the audit_log filtered by `incident_id`. There is no separate `incidents.chronology` table to drift from the canonical record. (ADR-025 + ADR-026 forensic-split inheritance.)
- **`DossierPreview` renders the exact dispatched bytes.** The preview is a PDF viewer, not a re-render from source data. This guarantees that what the operator sees on this surface = what the regulator received. A re-render would risk subtle divergence if templates change.

## Notes for implementation

- The PDF preview uses the browser's native PDF rendering (`<embed>` or `<iframe>` with `Content-Type: application/pdf`). No third-party PDF viewer JS. Fallback to plain-text rendering on browsers that block embedded PDFs (some mobile Safari configurations).
- The chronology rail is implemented as a React Suspense-streaming list — first paint shows the first 20 events, scroll triggers fetch of older. The audit_log query is indexed on `(organization_id, aggregate_id)` per ADR-031 + Wave 1.19 audit-log-ui indexes.
- Addenda are stored as a `recall_addendum` row with `incident_id` FK + `audit_log_envelope_id` FK. The composer's confirm action writes both rows in a single transaction via `recall.attach-addendum` MCP capability.
- The retry-status countdown polls the email-dispatch job status every 5 s when a retry is pending. After the third retry fails or after dispatch confirms, polling stops.
- First-paint < 1 s on laptop, < 1.5 s on phone. The dossier PDF can lazy-load below the fold; the receipt card + chronology rail render first.
- The breadcrumb (`Recall · Incidente IR-2026-0007 · Dossier`) shares the M1+M2 breadcrumb component. No new component.
- Accessibility: PDF preview includes a "Skip to dossier sections" link for screen readers; chronology rail has `role="log"` with `aria-live="polite"` so new events announce as they arrive.
