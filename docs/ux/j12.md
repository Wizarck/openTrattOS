# J12 — Operator reviews the Photo-Ingestion HITL queue

The Human-In-The-Loop review surface for vision-LLM extractions that fell into the **flag-for-review** confidence band (`0.60 ≤ confidence < 0.85` per ADR-034). The EU AI Act stance is that limited-risk AI MUST keep a human in the loop for consequential acts; this surface is where that loop lives. The visual surface is [mock-j12-photo-ingest-review.html](variants/mock-j12-photo-ingest-review.html); it anchors that contract to components.

## Goal

In ≤45 seconds per item, the operator (i) sees the photo + the LLM-extracted fields side-by-side, (ii) accepts or corrects each field with confidence colour-coding, and (iii) signs the ingest, producing either a Goods Receipt draft (invoice ingest) or a Lot creation (product ingest). Auto-fill items (`confidence ≥ 0.85`) skip this surface and land directly downstream; reject items (`< 0.60`) come here marked with a different visual treatment (more red, more required-fields).

## Trigger

Items land here from two paths: (i) Hermes invoke `inventory.ingest-invoice-photo` or `inventory.ingest-product-photo` from WhatsApp / Telegram / chat widget; or (ii) operator uploads photos directly from this surface's `+ Subir foto` CTA. The queue auto-opens when ≥ 1 item awaits review (per the user's role); otherwise it's reachable from the top nav.

## Walkthrough

The numbers below correspond to the regions of [mock-j12-photo-ingest-review.html](variants/mock-j12-photo-ingest-review.html).

1. **Header + transparency banner.** Eyebrow: `Revisión humana · ingestión por foto (HITL)`. Headline: `Cola de revisión · 4 elementos`. Below, a `--mute` paragraph: *"nexandro pide tu revisión humana sólo cuando la extracción del modelo está entre 60 % y 85 % de confianza. Por encima del 85 %, autocompleta; por debajo del 60 %, marca el campo para entrada manual. Esto cumple con el iron-rule HITL del EU AI Act."* The banner is load-bearing — operators trained on this rule make better calls.
2. **Queue list (left column, narrow).** Stacked rows, each = one item awaiting review. Row carries: thumbnail (left, 64 × 64) · type (`invoice` / `product`) · supplier or product hint · time-since-upload · confidence band badge. Selected row has a `--accent` left rule. Bottom of the list: `+ Subir foto` ghost CTA.
3. **Photo viewer (centre column).** The selected item's full-resolution photo, zoomable, with detected-field bounding boxes overlaid in `--accent`. Hovering / tapping a box highlights the corresponding extracted-field row on the right column. The viewer has a small toolbar: `+` zoom, `−` zoom, `↻` rotate, `↓` download original. Photo is always shown — operator review without seeing the source is exactly the failure mode HITL exists to prevent.
4. **Extracted fields (right column).** A vertical list of fields. Each row: field label (`Proveedor`, `Albarán nº`, `Línea 1 — producto`, `Línea 1 — cantidad`, `Línea 1 — precio unit`, `Total`, `Fecha entrega`) + extracted value (editable) + confidence badge per field. Confidence visual:
   - `≥ 0.85` (auto-fill band, rare on this surface but happens when only one field of many was flagged): `--success` dot + value in `--ink`.
   - `0.60 ≤ c < 0.85` (flag-for-review band, the typical case): `--mute` dot + value in `--ink` + small `revisar` ghost.
   - `< 0.60` (reject band, field required-by-operator): `--destructive` dot + value empty + `--destructive` border on the input + `Manual` mandatory eyebrow.
   Per-field tap → input becomes editable; the confidence badge converts to `editado por operador` once changed.
5. **Bounding-box ↔ field link.** Tapping a field row scrolls + highlights its bounding box in the viewer. Tapping a bounding box highlights its field row. The reciprocal link is load-bearing — without it the operator has to read the photo + the form simultaneously, a 2-second-glance bust.
6. **Confirm row.** `Cancelar` (ghost) + `Firmar ingestión` (primary `--accent`). The primary CTA disabled until all `<0.60` fields are filled. The confirmation writes an audit_log envelope with the original LLM extraction + the operator's corrections (both stored), then routes downstream:
   - `invoice` → creates a Goods Receipt draft in [J11](j11.md).
   - `product` → creates a Lot (which later receives CCP readings in [J10](j10.md)).
7. **Result interstitial.** Strip mounts: `✓ Ingestión firmada · GR draft creado · ver en Procurement →`. Two ghost actions: `Revisar siguiente` (advances to next queue item) and `Volver al panel`.
8. **EU AI Act provenance chip.** Bottom of the right column, a `--mute` line: `Modelo: gpt-oss-vision-72b · prompt v2.3 · confianza global 0.74 · audit_log AL-2026-189617`. Surfaces the AI provenance per FR41 and EU AI Act Article 13 (transparency). The chip links to the audit_log entry.
9. **Bulk review (chip group above queue).** Three chips: `Mis revisiones`, `Todas` (Manager+ scope), `Rechazadas`. The `Rechazadas` view shows past rejections — useful for spot-checking the vision-LLM's failure modes.

## Components used

`HitlQueueList` · `PhotoViewer` (zoomable + bounding boxes) · `ExtractedFieldList` · `ConfidenceBandBadge` (3 bands: auto-fill / flag-for-review / reject) · `FieldEditor` · `BoundingBoxOverlay` · `AiProvenanceChip`. See [components.md](components.md).

## Capabilities satisfied

FR28 (vision-LLM extraction with confidence), FR29 (HITL review on flag-for-review band), FR30 (auto-fill on ≥ 0.85), FR31 (reject on < 0.60 with operator-required fields), FR32 (extracted fields editable + original retained in audit_log), FR33 (route extracted data downstream — GR draft or Lot creation), FR41 (AI provenance disclosure per EU AI Act Article 13), NFR-OBS-2 (per-field confidence captured in audit_log). The boundary constants `>= 0.85` (auto-fill) and `>= 0.60` (flag-for-review) are code-level per ADR-034.

## Edge cases

- **Photo is unreadable / corrupted.** The viewer mounts a fallback `Imagen no se pudo cargar · re-subir →`. The queue row has the same fallback eyebrow. The operator can re-upload from the same row; the old photo is preserved in the audit_log envelope.
- **Vision-LLM extracted ZERO fields.** Overall confidence < 0.60. The row appears with a `--destructive` eyebrow `Extracción fallida · entrada manual completa · 7 campos requeridos`. The operator works as if uploading a fresh entry.
- **One field falls in two confidence bands across passes.** The system stores the highest-confidence pass; the operator never sees the lower one (would be confusing). The audit_log retains all passes for forensic review.
- **Operator corrects an auto-fill field after submission.** The correction creates a separate audit_log row (`HITL_RETROACTIVE_CORRECTION`). The original ingestion row is immutable.
- **Bounding box exists but the field label is unknown to the system.** The box mounts with a generic label `Campo desconocido (#N)` and an extracted value; the operator can either type the correct label or `Ignorar este campo`. The system learns from `Ignorar` decisions over time (per implementation notes).
- **Multiple operators view the same queue row simultaneously.** First operator to confirm wins; the second gets a `Otro operador firmó este ítem · 14:32 CEST por Carmen · ver resultado →`. No conflict.
- **Network drops mid-edit.** Local draft persists 30 minutes; on reconnect the operator resumes. Draft never enters audit_log.
- **Item came from Hermes but was sent to wrong namespace** (e.g. product photo classified as invoice). The viewer shows the type chip with a small `cambiar tipo →` ghost; the operator can re-classify. Re-classification produces a new audit_log row and re-routes downstream.

## Decisions specific to J12

- **Confidence-banded HITL is iron rule, never a configurable hint.** Per ADR-034: the boundaries `>= 0.85` (auto-fill) and `>= 0.60` (flag-for-review) are code-level constants, not operator-tunable. Letting operators lower the threshold opens a compliance gap (an operator under pressure could set the bar at 0.95, never trusting the system, OR at 0.30, trusting everything). Iron rule is the safety mechanism the EU AI Act expects (limited-risk + HITL by design).
- **Photo is ALWAYS shown alongside extracted fields.** The naive efficiency win — "the operator only needs to glance at the form" — is exactly what HITL is designed to prevent. The photo is the source of truth; the form is the model's hypothesis. The operator's job is to confirm hypothesis matches source. (PRD-M3 §Journey 3 visible-trust principle, locked.)
- **Bounding-box ↔ field reciprocal link is load-bearing.** Without it, the operator's eye has to scan-then-locate, twice. With it, attention transfers in one tap. (DESIGN.md §1 principle 8 — boring before clever; the affordance is conventional but valuable.)
- **Confidence visual uses dot + value styling, never only colour.** Per DESIGN.md §1 principle 5 + WCAG accessibility: colour alone cannot encode the band. The dot glyph + ghost-button cue + (for reject band) the destructive border give multiple cues. Operators with colour-vision differences still read the band.
- **Both LLM extraction and operator correction are stored.** The audit_log keeps the original alongside the correction. Inspectors and operators can later answer "what did the model think, and what did the human change?". This is the forensic foundation for tuning the vision-LLM prompt over time.
- **EU AI Act provenance chip surfaces the model + prompt version.** Per Article 13 transparency: operators (and downstream regulators reading audit_log) MUST be able to identify what produced the output. Burying this in metadata would meet the letter but miss the spirit; the chip is visible per item.
- **The `Rechazadas` chip surfaces past rejections.** Useful for spot-checking the vision-LLM and identifying systematic failure modes (e.g. "every Friday's invoices fail because the supplier prints in cyan"). The operator becomes a partner in tuning the model rather than just a passive gate.
- **Single sticky CTA: `Firmar ingestión` only.** No "Save partial review" — the entire review is one atomic act. Partial reviews invite "I forgot to check the last field" failure modes. The 30-minute local draft persistence protects against accidental loss; partial signing would be a confusing forensic story.
- **`+ Subir foto` CTA at bottom of queue list (not top).** The queue is the primary work; uploading is the secondary. Putting the CTA at the bottom of the list keeps the eye on the work, with the upload affordance reachable but not competing.

## Notes for implementation

- The vision-LLM call routes through `AiSuggestionProvider` DI per ADR-038 with the configured provider (`gpt-oss-vision-72b` default, Claude / GPT-4V available). The provider returns `extracted_fields[] { label, value, confidence, bounding_box }` + `overall_confidence`.
- Confidence boundary constants live at `apps/api/src/photo-ingestion/constants.ts`:
  ```
  export const CONFIDENCE_AUTO_FILL = 0.85;
  export const CONFIDENCE_FLAG_FOR_REVIEW = 0.60;
  ```
  Operators >= per ADR-034. Float-boundary tests verify IEEE 754 behaviour at the boundary.
- The bounding box overlay uses `<canvas>` with the photo as background and box rectangles drawn on top. Hover handlers link to field rows via component state (no DOM-querying).
- The queue list refreshes every 15 s via Server-Sent Events to surface new Hermes-seeded items live. Operator-action also pushes a refresh.
- Downstream routing on confirm:
  - `invoice` aggregate → POST to `procurement.create-goods-receipt` (draft state).
  - `product` aggregate → POST to `inventory.create-lot`.
  - Both write a `photo-ingestion.confirm-hitl-review` audit_log envelope first (chronologically pre-route).
- The "Rechazadas" view reads from audit_log filtered by `aggregate_type='photo_ingestion' AND payload_after.outcome='rejected'`.
- Draft persistence uses IndexedDB scoped to `(organization_id, hitl_review_id, actor_id)`. 30-minute TTL.
- First-paint < 1 s on laptop. The photo lazy-loads from object storage; the extracted-fields panel renders first.
- Accessibility: `PhotoViewer` provides a screen-reader description of each field's extracted value + confidence. Bounding boxes have `role="region"` + `aria-label` (e.g. "Campo Línea 1 — producto: extracted 'Atún rojo', confidence 0.91"). The CTA is `<button>`.
